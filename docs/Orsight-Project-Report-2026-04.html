Orsight（OrSight）项目开放进展报告

文档版本：2026-04-12
仓库：Orsight（双端 Web 应用：用户工作台 webapp + 管理端 admin-webapp）
说明：本报告基于当前代码库结构与近期迭代内容整理，用于对内/对外同步产品能力、阶段与成果。

一、项目概述

1.1 产品定位

Orsight 面向 物流 POD（Proof of Delivery）签退截图 等场景的 批量识别与结构化填表：用户上传多张截图或文档，系统结合 大模型视觉/文本能力 与 可配置表格字段，将信息抽取为可编辑表格记录，并支持 导出、人工复核、训练样本沉淀 等闭环能力。

1.2 仓库组成

模块 路径 角色
用户工作台 webapp/ 主业务：上传、识别、结果表、表单与训练标注、账户等
管理后台 admin-webapp/ 运营/管理：用户、可视化等（与主站分离部署）
训练与规则数据 training/（若存在）及按表单分目录的资源 训练池样本、全局规则等（由 API 与文件系统协同）

1.3 核心技术栈

  • 框架：Next.js App Router（webapp 当前 next@16.1.6，admin-webapp 略新小版本）
  • UI：React 19、Tailwind CSS v4
  • 认证：Supabase Auth（@supabase/ssr、@supabase/supabase-js），并保留开发环境会话/登录辅助接口
  • 文档与表格处理：xlsx、pdf-parse / pdfjs-dist、mammoth、word-extractor、sharp 等
  • 国际化：webapp 内中英双语资源（src/i18n/locales/zh.ts、en.ts）与 LocaleProvider

二、开放阶段与总体进度（定性）

2.1 阶段划分（建议表述）

阶段 状态 说明
MVP：识别 → 表格 → 导出 已具备 上传、抽取、结果编辑、分组与筛选、Excel 导出等主路径可用
多表单 / 多租户表单 已具备 表单列表、创建、复制、软删除与恢复、按 formId 隔离 API 与数据路径
训练闭环 已具备 标注弹窗、框选、训练池保存、预览试填、训练状态查询等
体验与可运营性 持续迭代 品牌与排版、中英文展示一致性、布局可调、标注交互优化等
企业级治理 部分具备 登录与权限依赖 Supabase；管理端独立；更细粒度审计/配额等需按产品路线扩展

2.2 当前开放进度（摘要）

  • 主业务链路：从「选表单 → 上传 → 识别 → 表格编辑 → 导出/标记正确」已形成闭环。
  • 训练增强链路：支持将人工框选与正确值写入训练池，并在后续识别中作为上下文与规则输入（具体策略见 lib/training 与相关 API）。
  • 工程成熟度：webapp 可通过 npm run build 生产构建；API 路由覆盖提取、表单、表格字段、训练子域等。

三、已实现功能（按域详细列举）

3.1 用户工作台（webapp）

3.1.1 首页 / 工作台（/）

  • 多文件上传：拖拽、点击选择、剪贴板粘贴（图片）；支持常见图片格式与 PDF 等（与提取管线一致）。
  • 批量识别与进度：抽取任务进度展示；主模型与复审模型分工（代码中可见 gpt-5-mini / gpt-5 等配置）。
  • 结果表：按表格项目（列）动态渲染；支持路线分组、筛选、行级操作；与「待复核/问题」联动展示。
  • 图片查看：记录关联多源图时的画廊查看、缩放与拖拽浏览（独立查看器区域）。
  • 人工标注入口：从记录进入 图片框选标注 弹窗，用于训练或纠错场景。
  • 布局与分区：上传区、结果与提醒区等支持用户可调尺寸（面板宽度/高度记忆等，见页面内状态与 localStorage 键）。
  • 导出：Excel 导出（含表头与行数据构建逻辑）。

3.1.2 表单首页（原「填表池」概念，现为「首页/Home」语义）

  • 表单卡片列表：创建、重命名、克隆、删除（默认模板策略与权限边界由 API 保障）。
  • 模板展示：系统模板名称/描述在英文界面下对 已知中文模板 做展示层映射（用户自定义名称保持原样），降低英文用户认知成本。
  • 进入配置与填表：从卡片进入表单配置流程或主填表界面（与路由参数 formId 绑定）。

3.1.3 表单配置 / 训练准备（FormSetupFlow / FormSetupClient 等）

  • 表格项目管理：启用/停用字段、内置字段与自定义字段等（与 table-fields API 协同）。
  • 从图片/文档推断列：模板来自截图识别、Excel 导入、文档推断等多来源（具体能力分散在对应 API 与组件流程中）。
  • 训练标注：在配置流程中打开与主工作台同源的 TrainingAnnotationWorkbench，用于沉淀训练样本。

3.1.4 训练模式页（/training）

  • 训练池图片网格：待标注列表与已标注状态展示。
  • 与表单字段联动：标注时使用当前表单激活字段定义。
  • 保存样本：写入训练池并刷新训练状态。

3.1.5 账户与登录

  • 登录页、账户页、OAuth 回调（auth/callback）等与 Supabase 流程衔接。
  • 开发辅助登录/会话（api/auth/dev-*）：便于本地与测试环境调试（生产部署需按安全策略收敛）。

3.1.6 图片框选标注（TrainingAnnotationWorkbench）

  • 双模式：单条记录 / 完整表格模式切换；与后端 preview-fill 的自动判定结果对齐。
  • 字段选择与画框：字段与填写区合并为同一工作区；支持多框、数字字段默认合并策略（求和/拼接等）。
  • 缩放与平移：放大缩小；拖动画面 模式下在视口内平移查看（含横向/纵向可拖动画布区域）。
  • 左右分栏可调：桌面端拖拽中间分隔条调节图片区与表单区比例；窄屏纵向堆叠时分隔条调节上下比例。
  • AI 试填预览：调用 /api/training/preview-fill，将框选区域送模型试填并写回右侧表单。
  • 存入训练池：调用 /api/training/save，持久化样本、框与聚合配置等。
  • 确认填入主表（不存训练池）（主工作台场景）：通过 onApply 将当前（含试填后核对过的）值写回首页结果行，不调用训练池保存接口，满足「只纠错入库、不污染训练集」的场景。

3.2 后端 API 能力（webapp/src/app/api）

路径 能力摘要
POST /api/extract 核心抽取：多文件、表单上下文、vision 上下文、问题列表与记录结构化输出
GET/POST /api/forms 表单列表加载、创建等
POST /api/forms/[formId] 更新、复制、软删除/恢复、永久删除（含密码校验分支）、标记 ready 等
POST /api/forms/template-from-image 从模板图生成/辅助表单结构
GET/POST /api/table-fields 读取/保存表格字段配置
POST /api/training/save 保存训练样本（输出、框、表格输出、模式等）
POST /api/training/preview-fill 基于框选与字段定义的试填预览
GET /api/training/status 训练池/样本量等状态
GET /api/training/image 按图片名读取训练相关图像数据
GET/POST /api/training/rules 训练规则读取/更新
POST /api/training/guidance-chat 训练/配置向导类对话接口
POST /api/training/parse-document 文档解析分支（与训练/文档上传场景相关）
GET /api/training/context-asset 训练上下文资源读取
dev-login / dev-logout / dev-session 开发环境认证辅助

3.3 管理端（admin-webapp）

  • 登录与受保护首页、用户列表与详情、可视化（viz） 等页面骨架存在，用于运营侧数据查看（具体权限与数据范围依赖后端配置）。

3.4 工程与体验规范

  • 字体栈统一：Noto Sans + Noto Sans SC，且 字体 CSS 变量挂在 <html> 以兼容 Tailwind v4 preflight（见 .cursor/rules/orsight-typography.mdc）。
  • 顶部品牌：用户端导航展示产品名 Orsight。
  • i18n：关键界面字符串中英双语；动态数据（系统模板名、内置列标题等）通过 form-display、table-field-display 等工具在英文环境下做 展示层 映射，不强制改写用户自定义内容。

四、解决的典型场景问题（问题 → 方案）

4.1 识别结果与表格结构不一致

  • 问题：不同客户表单列不同，硬编码列无法满足。
  • 方案：表格项目可配置 + 抽取 API 按激活字段集合构建 prompt 与校验逻辑。

4.2 英文用户看到大量中文系统文案

  • 问题：模板名、内置字段标签等来自中文默认值，切换英文后体验割裂。
  • 方案：i18n 词条 + 对 系统模板/内置字段 的展示映射；用户自定义名称保持原样。

4.3 字体未按预期生效（看起来像系统默认字体）

  • 问题：Tailwind v4 在 html 上应用 font-family，而 next/font 变量只在 body 上时，变量无法作用于 html。
  • 方案：将字体 variable class 提升到 <html>，并在 globals.css 明确 html { font-family: ... }；文档化到 Cursor Rule 防止回归。

4.4 标注时「填值」与「选字段画框」割裂

  • 问题：操作路径长、易错。
  • 方案：合并为单一工作区，每字段旁选择当前画框目标，保留多框与聚合逻辑不变。

4.5 标注弹窗底部操作混乱

  • 问题：附加说明、重复关闭按钮、主操作不醒目。
  • 方案：移除冗余区块；主按钮（试填/保存）跨弹窗底部居中；保留 Esc/右上角关闭符合习惯。

4.6 图片区滚动与「拖动画面」预期不一致

  • 问题：竖图在常用缩放下横向可拖范围不足，用户感觉只能纵向动。
  • 方案：为画布外包一层可滚动平面并预留 padding，使平移有明确横向行程；移除底部「主区/页脚」高度拖拽条，避免与「拖动画布」混淆。

4.7 试填结果只停在弹窗内，无法只更新主表

  • 问题：用户确认预览正确但不想写入训练池。
  • 方案：新增 「确认填入主表（不存训练池）」，走 onApply 回调更新首页记录；「存入训练池」 仍负责样本沉淀并可继续同步主表。

五、开放成果（可对外表述的要点）

  1. 端到端工作台：上传 → 识别 → 编辑 → 导出，支持多表单隔离。
  2. 可演进字段模型：内置 POD 相关字段 + 自定义列，适配不同客户表格。
  3. 训练与试填闭环：框选 + 正确值 + 保存样本 + 预览试填，提升长尾截图适配速度。
  4. 双语与品牌化：统一字体与导航品牌，系统内容英文化展示策略清晰。
  5. 工程可交付：Next.js 16 + React 19 + TS，构建与路由结构清晰，API 面完整。

六、风险、边界与后续建议

6.1 模型与成本

  • 识别质量与费用受模型版本、调用量、图片分辨率影响，需在生产环境配置 限流、缓存、失败重试与监控。

6.2 数据与合规

  • 截图可能含个人信息；需明确 留存周期、访问控制与审计（依赖 Supabase 策略与产品政策）。

6.3 训练数据质量

  • 「确认填入主表但不存训练池」可避免污染训练集；同时应配套 样本审核、去重与版本化 策略（部分已在训练管线中体现，可继续产品化）。

6.4 建议的下一里程碑（可选）

  • 管理端与主站 统一版本策略（两端 Next 小版本差）。
  • 标注历史/版本回滚（行级变更记录）。
  • 批量试填/批量应用 与 差异高亮（提升运营效率）。
